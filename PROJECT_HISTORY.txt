Current Data Setup
------------------
- For now the app uses db.json as the internal database.
- In JSON mode, `db.json.users` is app data and not the active password source for login.
- `admin / admin` currently works because the local JSON store has a demo login fallback in code.
- SQL Server support has now been added for local SSMS / SQLEXPRESS use.
- If SQL env vars are not set, the app still falls back to `db.json`.
- `server.js` now supports loading a local `.env` file automatically.
- Current SQL path expects:
  - server: `localhost`
  - database: `ScheduleTracker`
  - SQL login: `scheduleapp`
  - SQL password: `Schedule@123` (or the user-chosen replacement)
- Current blocker found on this machine:
  - the app now reaches SQL Server successfully with `scheduleapp`
  - the first `/api/bootstrap` call fails with: `CREATE TABLE permission denied in database 'ScheduleTracker'`
  - `scheduleapp` needs `db_ddladmin`, or `dbo.schedules` must be created manually by an admin user
- SQL login behavior:
  - JSON mode uses a development/demo login path
  - SQL Server mode is intended to use application users stored in `dbo.users`
  - `admin / admin` should be treated as a dev fallback, not as the real database user account

Current SQL App Mapping
-----------------------
- `Schedules` component -> `dbo.employeeSchedule`
- `Projects` component -> `dbo.employeeProject`
- `Calender` component -> `dbo.employeeCalendar`
- `Meetings` component -> `dbo.employeeMeeting`
- `Organization` component -> `dbo.users`

Current SQL Application Flow
----------------------------
- `Login Page -> dbo.users -> session identity -> API calls -> feature tables -> dashboard UI`
- Login checks:
  - `EmpUsername`
  - `EmpPassword`
  - `EmpID`
  - `EmpFullName`
  - `EmpDesignation`
- After login, the app should use the logged-in employee identity for all feature fetch/save calls.
- Current intended module flow:
  - `Schedules` reads and writes `dbo.employeeSchedule`
  - `Projects` reads and writes `dbo.employeeProject`
  - `Calender` reads and writes `dbo.employeeCalendar`
  - `Meetings` reads and writes `dbo.employeeMeeting`
  - `Organization` reads `dbo.users` and builds the tree from `EmpReportingManagerID -> EmpID`
- Recommended future access scope:
  - employee -> own data
  - manager -> own data plus junior data
  - owner/admin -> all data
